# この時間のおすすめ曲 仕様

## 概要

トップページのタイムライン上部に、`この時間のおすすめ曲` を 1 曲表示する。

目的:

- 投稿された曲の中から、まず 1 曲聴いてもらう導線を作る
- タイムライン全体の再生数を増やす
- サイトに「今ここで聴く理由」を持たせる

このおすすめ曲は、タイムライン内の曲を対象に、`いいね率` をもとに選出する。

## 表示方針

- 表示位置は `TIMELINE` セクションの先頭
- 見出しとして `この時間のおすすめ曲` を表示する
- 必要に応じて小さく `HOURLY PICK` を添える
- おすすめ曲は 1 曲のみ表示する
- 通常の曲カードと同じ UI 構造をベースにする
- ただし、おすすめ枠全体は背景色や枠線を少し変えて、特別感を持たせる
- 既存デザインから大きく外れないことを優先する

## 更新タイミング

厳密な cron は使わず、無料の Render Web Service 上で動かす。

更新の考え方:

- 毎時ちょうどに自動実行するのではなく、`誰かがトップページを開いた結果として API が呼ばれたとき` に更新判定を行う
- サーバー側で `時間キー` を持つ
- 時間キーが変わっていた場合のみ、その時間帯のおすすめ曲を再計算する
- 同じ時間帯なら再計算しない

例:

- 保存済み時間キー: `2026061409`
- 現在の時間キー: `2026061410`
- 異なるため、10時台のおすすめ曲を再計算して保存する

## 時間キー

時間キーは以下の形式とする。

```text
YYYYMMDDHH
```

例:

- `2026061409`
- `2026061410`

用途:

- この時間帯ですでにおすすめ曲を確定したかどうかの判定に使う

## 起動トリガー

起動トリガーは、ユーザーがトップページを表示した際にフロントエンドから呼ばれる API とする。

想定:

- `おすすめ曲取得 API`
- または `タイムライン取得 API`

どちらでも実装可能だが、責務の分離を考えると専用 API のほうが分かりやすい。

## 選曲対象

対象は、`現在タイムラインに存在している曲` のみとする。

除外条件:

- 通報中の曲 (`reportActive = true`)

候補条件:

- `playCount >= 2`

補足:

- 24 時間で消える仕様のため、タイムライン内の曲はすべて有効な候補母集団とみなしてよい

## 選曲ロジック

おすすめ曲は `いいね率` をベースに決定する。

単純な `likeCount / playCount` だと、再生数が極端に少ない曲が不自然に強くなるため、以下の補正式を使う。

```text
score = likeCount / (playCount + 3)
```

意図:

- 再生数が少なすぎる曲の過大評価を防ぐ
- それでも、ちゃんと評価されている曲は上に来る

## 並び順

おすすめ曲の決定順は以下とする。

1. `score` 降順
2. `createdAt` 昇順
3. `id` 昇順

意味:

- まず `いいね率` が高い曲を優先
- 同率なら、より古くから流れている曲を優先
- さらに同一なら `id` で安定化する

SQL イメージ:

```sql
ORDER BY
  score DESC,
  created_at ASC,
  id ASC
```

## フォールバック

候補が 1 曲もない場合は、以下の順でフォールバックする。

1. 通報中でないタイムライン曲のうち、最も新しい曲を表示する
2. それも存在しない場合は、おすすめ枠自体を非表示にする

補足:

- `playCount >= 2` を満たす曲がなくても、おすすめ枠が空になる時間帯を減らすため、新着フォールバックを用意する

## 表示内容

おすすめ曲枠には以下を表示する。

- ラベル: `この時間のおすすめ曲`
- 必要なら英字補助: `HOURLY PICK`
- ジャケット画像
- 曲名
- アーティスト名
- `PLAY`
- `OPEN IN SUNO`

通常カードと差をつける点:

- おすすめ枠の背景色を少し変える
- 枠線を通常より少し強める
- 上下余白を少し広げる

## デザイン方針

おすすめ枠は、`別物の派手なカード` ではなく、`通常カードの少し特別版` とする。

やること:

- タイムライン先頭に置く
- 背景色をわずかに変える
- 枠線を少しだけ強める
- ラベルを追加する

やらないこと:

- ヒーロー並みに大きくする
- 強いグラデーションにする
- 金色や王冠などの強い演出を入れる
- 既存 UI と別デザインにしすぎる

キーワード:

- 静かな特集枠
- 少し格上
- 既存デザインとの連続性

## サーバー側の保持情報

最低限、以下を保持する。

- `recommended_hour_key`
- `recommended_track_id`
- `recommended_selected_at`

必要に応じて以下も持てる。

- `recommended_score`

## 動作フロー

1. クライアントがトップ表示時におすすめ曲取得 API を呼ぶ
2. サーバーが現在時刻から `hourKey` を生成する
3. 保存済み `recommended_hour_key` と比較する
4. 同じなら保存済み `recommended_track_id` を返す
5. 違うならおすすめ曲を再計算する
6. 計算結果を保存する
7. 保存したおすすめ曲を返す

## 期待する効果

- 最初の 1 曲を再生する導線ができる
- タイムライン全体の曲が聴かれやすくなる
- 一覧から選ぶ負担を減らせる
- サイトに「今のおすすめ」が生まれ、訪問理由になる

## 将来拡張案

今後必要なら、以下に拡張できる。

- おすすめ履歴の保存
- 時間帯別のラベル変更
  - 例: 深夜のおすすめ、昼のおすすめ
- 手動ピックとの併用
- 再生数の勢いも加味した複合スコア化
