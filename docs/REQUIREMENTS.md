# SUNO Timeline 要件整理

## 1. コンセプト

`SUNO Timeline` は、Suno の曲だけが静かに流れていく、最小限の匿名音楽タイムラインです。

目指すのは、上品で美しく、無駄を極力削った体験です。

- 名前は出ない
- プロフィールもない
- コメントもない
- 誰の投稿かも分からない
- 誰がいいねしたかも分からない

ただ曲が並び、気になった曲を再生して、好きならいいねを押すだけのサイトにします。

## 2. デザイン方針

- 白を基調とした、静かで洗練された見た目
- 情報量を絞ったミニマルな1ページ構成
- 主役は曲そのもの
- SNS 的な装飾は置かない

置かないもの:

- アイコン
- ユーザー名
- プロフィール
- フォロー
- コメント
- 投稿者情報
- 通知

## 3. ユーザーができること

ユーザーが行う行動は、基本的に次の4つです。

1. ページを開いてタイムラインを見る
2. 曲を再生する
3. 曲にいいねする
4. Suno のリンクを貼って曲を投稿する

## 4. サイトの基本ポリシー

- ログインなし
- アカウントなし
- 投稿者の概念なし
- いいねした人の概念なし
- 匿名で使える
- ただしサイト自体は共有タイムラインとして機能する

## 5. 共有サイトとしての方針

今後は、各ブラウザの中だけで完結するローカル保存ではなく、訪問者全員が同じ投稿一覧を見る共有サイトにします。

つまり:

- ある人が曲を投稿したら、他の人にもその曲が見える
- ある人がいいねしたら、いいね数は全体に反映される

一方で、次の情報は共有しません。

- 誰が投稿したか
- 誰がいいねしたか

共有されるのは、曲そのものと、いいね数だけです。

## 6. 保存方式

共有データの保存先には `SQLite` を使います。

理由:

- 軽量
- 無料で使える
- 小規模な共有サイトに向いている
- 大げさな外部DBを用意しなくてよい

## 7. 投稿するSunoリンクの対応範囲

投稿時には、Suno の共有リンクを受け取り、正しい埋め込みプレイヤーへ変換できるようにします。

対応対象:

- 通常の曲URL  
  例: `https://suno.com/song/{uuid}`

- `?sh=...` が付いた曲URL  
  例: `https://suno.com/song/{uuid}?sh=...`

- Share → Copy Link で取得される短縮URL  
  例: `https://suno.com/s/{share_code}`

期待動作:

- 短縮URLでも正しく曲を特定できる
- 特定後は埋め込みプレイヤーを表示できる
- 無効なURLのときは分かりやすいエラーメッセージを出す

## 8. タイムライン仕様

- 投稿された曲は共有タイムラインに流れる
- 縦スクロールで曲を見ていく
- 各カードに表示する要素は最小限にする

各曲カードに含めるもの:

- 埋め込みプレイヤー
- Suno 原曲ページへのリンク
- いいねボタン
- いいね数

含めないもの:

- 投稿者名
- 投稿者ID
- 説明文
- コメント

並び順:

- ひとまず新しい投稿が上に来る

## 9. いいね仕様

いいねは匿名のまま扱いますが、完全な無記録にはしません。

### 必要な動作

- ユーザーは曲にいいねできる
- 同じユーザーは、その曲のいいねを取り消せる
- いいね数は全員共通で見える
- 自分が押したかどうかの状態は、同じブラウザでは保持される

### 採用する考え方

ログインは導入しません。
その代わり、ブラウザごとに匿名IDを発行して扱います。

動きとしては:

- 初回アクセス時に匿名IDを生成
- その匿名IDをブラウザに保存
- いいね時にその匿名IDを一緒に送る
- サーバー側では `track_id + anonymous_client_id` で管理する

これにより:

- 人物の正体は分からない
- でも「このブラウザのこの匿名ユーザーが押した」は判定できる
- 同じブラウザで見返したときに押下状態を復元できる

## 10. 匿名IDの扱い

匿名IDは、公開プロフィールのようなものではありません。
あくまで以下の目的だけで使います。

- いいね状態の保持
- 同一ブラウザからの重複いいね制御

匿名IDは `localStorage` などのブラウザ内保存で持たせる想定です。

## 11. 非目標

今回の段階では、次の機能は対象外とします。

- ユーザー登録
- ログイン
- コメント
- 検索
- タグ
- フォロー
- 通知
- 管理画面
- 投稿者プロフィール

## 12. バックエンドの役割

バックエンドは少なくとも次の役割を持ちます。

- Webページを配信する
- Suno の短縮リンクを解決する
- 投稿曲を SQLite に保存する
- いいね情報を SQLite に保存する
- 共有タイムラインを返す
- 現在の匿名IDに対する「押した状態」を返す
- いいね / いいね解除を受け付ける

## 13. 想定データ構造

### tracks テーブル

- `id`
- `source_url`
- `canonical_url`
- `embed_url`
- `track_key`
- `created_at`

### likes テーブル

- `track_id`
- `anonymous_client_id`
- `created_at`

制約:

- `track_id + anonymous_client_id` は1件だけ

## 14. 想定API

実装上は、少なくとも次のようなAPIが必要です。

- 曲投稿
- タイムライン取得
- Sunoリンク解決
- いいね
- いいね解除

タイムラインで各曲に含めたい情報:

- 曲情報
- 共有いいね数
- 現在の匿名クライアントが押しているかどうか

## 15. 現時点の合意内容まとめ

ここまでで合意している方向性は次の通りです。

- サイトは匿名のままにする
- ただし投稿内容は全員に共有される
- 保存先は SQLite にする
- Suno の通常URLと短縮共有URLの両方に対応する
- いいねは匿名IDベースで扱う
- 1ブラウザ単位で押下状態を保持する
- 曲以外の情報は極力見せない
- ミニマルで上品な1ページ体験を維持する

## 16. 次の実装方針

次に進める実装は、現在のブラウザ内ローカル保存中心の状態から、SQLite を使った共有タイムラインへ移行することです。

進める内容の中心は以下です。

- SQLite スキーマ作成
- 投稿保存のサーバー化
- タイムライン取得API
- 匿名IDの発行
- いいねの永続化
- 押下状態の復元
