# SUNO Timeline 24 現在仕様メモ

最終更新: 2026-06-14
対象フォルダ: `E:\AI\Codex\SUNOタイムライン2`
本番URL: `https://suno-timeline-24.onrender.com/`

## 1. サイトの思想

- SUNO の曲を静かに流して聴ける場所にする
- 情報を増やしすぎず、曲そのものに意識を向ける
- 気になった曲は `OPEN IN SUNO` から元ページへ行けるようにする
- 投稿者プロフィールやSNS的な機能は持たせない
- 24時間で流れて消える、軽いタイムライン体験を大事にする

## 2. 技術構成

### フロントエンド

- `index.html`
- `styles.css`
- `app.js`

### サーバー

- `server.mjs`

## 3. 現在の主要仕様

### タイムライン

- 投稿された曲は新しい順で表示
- 投稿から24時間で自動削除
- 数秒おきに再取得してほぼリアルタイム反映
- 再生中でもタイムラインだけ更新できる

### 投稿

- SUNO の共有リンクのみ投稿可能
- `song/...` 形式と `s/...` 短縮形式に対応
- URL バリデーションあり
- 連投制限あり

### 再生

- 下部固定プレイヤーで1曲だけ再生
- 曲カードを押すとその曲を再生
- `OPEN IN SUNO` で元ページへ遷移可能

### いいね

- 全ユーザー共有カウント
- 同じブラウザからは1曲につき1回まで
- ブラウザ側の匿名IDで判定

### 超推し！

- 全ユーザー共有カウント
- 同じブラウザからはタイムライン全体で同時に1曲だけ
- トグル式で解除可能
- 曲がタイムラインから消えれば、そのブラウザは別の曲に超推しできる

### 通報

- トグル式
- 最後に ON になってから1時間 ON のままなら曲を削除
- OFF に戻すと通報タイマーはリセット
- 寿命メーターは「24時間」と「通報1時間」の短いほうを表示

## 4. 現在のUI仕様

### ヒーロー

- タイトルは `SUNO TIMELINE 24`
- サブコピーは `MUSIC STREAM`
- リード文は `それは24時間で消える魔法。`
- 右側にマスコット画像を表示

### 統計カード

- 現在の投稿数
- 現在のいいね数
- 現在の再生数
- 現在の超推し！数

### この時間のおすすめ曲

- タイムライン上の曲から毎時間1曲を選出
- 基本ロジック:
  - いいね率が高い曲を優先
  - 同率なら古い曲を優先
- タイムライン上部に特別枠で表示

### 超推し！ピックアップ

- `超推し！数 > 0` の曲だけを表示
- 並び順:
  - `YOUR PICK` が先頭
  - それ以外は新しい曲が上
- 初期表示は6件
- 7件以上あると `+n more` で展開
- 展開後は `Show less` で閉じる
- `VIEW` で元のタイムラインカードへスクロール

## 5. AUTO PLAY の現在仕様

### 通常 AUTO PLAY

- タイムライン順に上から再生

### AUTO PLAY (SHUFFLE)

- タイムライン曲をシャッフル再生

### AUTO PLAY (超推し！のみ)

- `超推し！数 > 0` の曲だけを順番再生
- 対象曲が1曲もない場合はボタンを無効化

### iPhone Safari の注意

- iPhone Safari は埋め込みプレイヤーの自動再生制約が強い
- この環境では想定通りに連続再生できないことがある
- そのため一部で手動シーケンス寄りの制御を使っている

## 6. 超推し！のDB仕様

### テーブル

- `super_likes`

### 考え方

- 1ブラウザが同時に持てる超推しは1件
- `anonymous_client_id` を主キーにして制御
- 解除するとそのブラウザは別の曲へ超推し可能

## 7. 削除と寿命

- 曲は投稿後24時間で物理削除
- 通報が1時間継続した曲も物理削除
- 削除された曲の `likes` と `super_likes` は外部キーにより追従削除される設計

## 8. 環境

- `main` へ push すると Render が自動デプロイ

## 9. DB反映の考え方

- 現在はアプリ起動時に `CREATE TABLE IF NOT EXISTS` や `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` を実行する方式
- そのため今回のような追加系変更は、GitHub に push して Render が新コードで起動すれば Supabase 側へ自動反映される
- ただし以下は注意:
  - 列名変更
  - データ移行
  - 既存値の再計算
  - 削除系変更
  こうした変更は将来的に専用 migration のほうが安全

## 10. リリース運用

### 基本フロー

1. ローカル修正
2. `127.0.0.1:4173` で確認
3. `git add`
4. `git commit`
5. `git push origin main`
6. Render の自動デプロイを待つ
7. 本番URLで確認

### 補足

- Render 無料枠はスリープするので初回アクセスが遅いことがある
- デプロイ直後は旧プロセスと新プロセスの切り替わりで数十秒ほど表示差が出ることがある
